Задача 3 - АПИ

Можно придумать любую сущность, буду писать на примере сущности WIsh (желание, типа мы приложуху со списком желаний делаем)

Сперва создадим таблицу в базе данных (согласно доке призмы), пусть минимальный набор колонок будет
id - можно выбрать самостоятельно какой будет айди, самые распространенные: int или uuid

А все остальные колонки - на вкус и цвет, можно баловаться с бд здесь, например создать еще табличку с категориями и присваивать желаниям категорию или категории (будут  разные виды связи) и т.д.

Затем надо разработать саму апи
Если сервак на несте, настоятельно рекомендую использовать cli для генерации крудов, например nest g res wish (https://docs.nestjs.com/cli/usages)

https://restfulapi.net/ - тут можно почитать про конвенции реста и быть душнилами

CRUD - Create Read Update Delete

Create - ендпоинт для создания желания
Семантически этот ендпоинт должен использовать метод POST (никто не запрещает использовать хоть метод KAPUT, но по правилам при создании чего-то используется POST)
Требования к ендпоинту
Принимать какой-то пейлод, который будет соответствовать модели желания
Валидировать этот пейлод, например, что название должно быть обязательным и содержать не больше 255 символов (это если мы, например, используем varchar(255) для его хранения в базе)
Для неста можно почитать тут гайдик по декорированию+валидированию https://docs.nestjs.com/techniques/validation#using-the-built-in-validationpipe
Создавать запись в бд в соответствующей табличке
В случае успеха отдавать респонс со статусом 201 Created (это по семантике, можно какой угодно), для ошибок использовать соответствующие статусы ошибок, например 400 Bad Request для ошибок валидации
Read - ендпоинт для чтения, мы сделаем 2
Семантически используем метод GET, т.к. мы запрашиваем данные от сервера
Read List - вывод списка
Требования к ендпоинту
Принимать параметры пагинации и фильтрации (любой вообще, например поиск), можно так же сделать параметры сортировки
Валидировать параметры необязательно, но можно, если мы хотим, чтобы пагинация передавалась всегда, например
В случае успеха отдавать респонс со статусом 200 OK (это по семантике, можно какой угодно), для ошибок использовать соответствующие статусы ошибок, например 400 Bad Request для ошибок валидации
Read One - вывод одного конкретного желания
Требования к ендпоинту
Принимать айди записи, которую мы хотим достать
Можно валидировать айди на соответствие типу данных, например, если в качестве айди используется uuid - то его легко проверить, так же если айди это число - то мы можем проверять что нам приходит именно интовая цифра и т.д.
В случае успеха отдавать респонс со статусом 200 OK (это по семантике, можно какой угодно), для ошибок использовать соответствующие статусы ошибок, например 400 Bad Request для ошибок валидации
Update - ендпоинт для обновления желания
Семантически этот ендпоинт должен использовать метод PATCH (если мы обновляем только то, что отправляем) или PUT (если мы обновляем всё что отправили, а оставшееся дефолтим, обычно в null), можно сделать какой угодно
Не забываем про айди записи, которую мы обновляем, часто прежде чем делать запрос в базу на обновление проверяется есть ли такая запись в таблице (если нет - 404 Not Found)
Требования к ендпоинту
Принимать айди в виде параметра и какой-то пейлод обновления, который будет соответствовать модели желания
Валидировать этот пейлод, например, что новое название должно содержать не больше 255 символов
Обновлять запись в бд в соответствующей табличке согласно семантике HTTP метода
В случае успеха отдавать соответствующий респонс и так же для ошибок
Delete - ендпоинт для удаления желания
Семантически этот ендпоинт должен использовать метод DELETE
Требования к ендпоинту
Принимать айди записи для удаления
Обновлять удалять записть в соответствующей табличке
В случае успеха отдавать соответствующий респонс и так же для ошибок



Домашка по бэкенду 1
Условия
Сервер должен быть написан на JS или TS
Сервер должен работать в рантайме node.js >=22.0.0
Менеджер пакетов должен быть npm
Сервер должен запускаться командой npm run start
Фреймворк может быть одним из следующих
Nest.js - NestJS
Express - Express.js

Задача 1
Поднять сервер!
Суть задания в том, чтобы предоставить репозиторий (через git или zip архив, как угодно), в котором будет
Файловая структура, содержащая в корневой директории как минимум следующие файлы и папки
/src - директория для всей логики приложения
/src/main.js (или .ts) - файл, откуда запускается сервер
package.json
package-lock.json
.env.example - файл содержащий названия переменных окружения, но не значения
В main файле будет запускаться сервер на порту, указанном в environment (например в переменной PORT)

Задача 2
Сделать ендпоинт
Нужен ендпоинт, для проксирования запросов в API https://artsearch.io/ с фронта (вообще можно любую апи использовать)
Критерии приёмки:
API ключ хранится в env проекта и название переменной добавлено в .env.example
Ендпоинт доступен без аутентификации
Ендпоинт возвращает оригинальную структуру ответа от внешней API
Ендпоинт возвращает 418 http статус если сторонняя апи возвращает ошибку
Метод/функция, исполняемый при вызове, использует
HttpService, если используется Nest.js
Axios, если используется express.js

Задача 3 (опционально)
Универсальное логирование входящих запросов
Очень важно подготовить себе инструменты трейлинга и дебаггинга, поэтому задача - создать универсальный способ логировать любой входящий на сервер http запрос.
Критерии приёмки:
Разработан middleware для логирования входящих запросов в консоль (console.log)
В логе должны быть
Полный url запроса
HTTP метода запроса
Таймстамп запроса
Какой бы ендпоинт мы ни добавили - запросы на него будут логироваться благодаря разработанному Middleware
